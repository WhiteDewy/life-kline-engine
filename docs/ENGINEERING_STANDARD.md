# 工程规范 · Engineering Standard (v1.0)

> **本文件是本项目的最高开发准则。** 所有开发者与所有 AI subagent 在提交任何代码前，必须确认改动符合本规范。与本规范冲突的旧代码视为技术债，须按“迁移要求”逐步偿还，不得新增同类违规。
>
> 定位：**企业级、工程化、需落地、承接高并发**。判断一切取舍的最高标准是：**在真实用户量与并发压力下，系统是否可靠、可观测、可扩展、可维护，且守住疗愈产品的伦理底线。**
>
> 关键词分级：**MUST**（必须，违反即阻断合并）／**SHOULD**（应当，偏离需在 PR 说明理由）／**MAY**（可选）。

## 1. 架构与代码组织

- **MUST** 分层清晰，依赖单向：`路由层(router) → 服务层(service) → 领域引擎(life_kline) → 基础设施(db/llm/cache)`。上层可依赖下层，下层禁止反向依赖。
- **MUST** 路由函数（FastAPI handler）保持“薄”：只做参数校验、鉴权、调用服务层、组装响应。**禁止**在 handler 内写业务逻辑、直接拼 SQL、直接调 LLM。
- **MUST** `backend/main.py` 停止增长。新端点一律放入 `backend/routers/<domain>.py`，用 `APIRouter` 挂载。（偿还 D4）
- **MUST** 领域引擎（`src/life_kline/`）保持纯粹：不依赖 FastAPI、不读环境变量、不碰 HTTP。引擎是可单测的纯函数/纯类。
- **MUST** 遵循 CLAUDE.md 既有铁律：**引擎逻辑必须规则驱动、通用化，禁止为特定人物写硬编码分支。**
- **SHOULD** 单文件不超过 ~600 行、单函数不超过 ~50 行；超出考虑拆分。
- **SHOULD** 公共能力（LLM、缓存、限流、鉴权）沉淀为独立模块，供两套对话系统（星灵 / 星语者）复用，避免重复实现。

## 2. 并发模型（高并发地基）

- **MUST** 服务无状态。任意请求可路由到任意实例，实例可随时水平扩缩容。所有会话/对话状态外置到 Redis/DB（见 §4）。（偿还 D2）
- **MUST** **禁止在 async 路径中出现阻塞调用**：包括同步 HTTP（`urllib`/`requests`）、同步文件 IO、`time.sleep`、阻塞的 DB 驱动。（偿还 D1）
  - 外部 HTTP 用 `httpx.AsyncClient`（或 aiohttp）。
  - 不可避免的 CPU 密集/阻塞库（如 `pyswisseph` 星历计算）**MUST** 用 `run_in_executor` / `anyio.to_thread` 丢到线程池，且**必须**带超时。
- **MUST** 全局复用连接：HTTP client、DB 连接池、Redis 连接池在应用启动时创建、`lifespan` 中关闭，**禁止**每请求新建。
- **SHOULD** 星历/评分等重计算结果按输入（出生时刻+经纬度+房屋制）缓存，避免重复算盘。

## 3. 外部依赖与 LLM 调用规范

> LLM 是本产品最慢、最不稳定、最贵的依赖，单列铁律。

- **MUST** LLM 调用全链路异步、非阻塞（偿还 D1、D5）。提供 `chat()`（一次性）与 `chat_stream()`（流式）两套异步接口。
- **MUST** 分级超时：连接超时 ≤ 3s；首 token 超时 ≤ 10s；整体超时按场景配置。超时**必须**可通过配置调整，不得硬编码。（偿还 D10）
- **MUST** 重试仅针对**幂等且瞬态**的失败（网络、5xx、超时），带指数退避 + 抖动，**最多 2 次**；4xx（鉴权/参数）**禁止**重试。
- **MUST** 熔断/降级：LLM 不可用时，**必须**降级到引擎原文（本项目已有的 `mirroring`/`fallback` 机制），**绝不**因 LLM 挂掉而使整个对话 500。
- **MUST** **防幻觉红线（疗愈产品核心）**：星盘/事实类数据**只能来自引擎或工具返回**。工具失败时，System Prompt 必须指令模型**如实说“暂时看不清这部分”**，**严禁**编造星盘数据。工具异常须捕获并**转成自然语言**回灌给模型，而非抛代码错误。
- **MUST** 流式实现“垫场句”：触发工具/长计算时，立即先推一句共情文本，后台执行，返回后再续流——**用户永远不干等**。
- **MUST** 客户端断连时，**必须**能取消后台 LLM 请求，释放连接与配额，防止资源泄露。
- **MUST** provider 无关：所有 LLM 调用经统一抽象层，切换 DeepSeek/Claude/Qwen 只改配置。（现有 `llm_client` 已具雏形，须补齐异步与流式。）
- **SHOULD** 记录每次调用的 token 用量、耗时、provider、是否降级，用于成本与性能分析。

## 4. 状态与存储（单一可信源）

- **MUST** **单一可信源（Single Source of Truth）**：同一份数据只有一个权威存储。消除 JSON 文件 + SQLite 双写。（偿还 D3、D2）
  - 会话/短期对话状态 → **Redis**（带 TTL）。
  - 持久业务数据（用户、报告、日记、消息、消费）→ **关系数据库**（当前 SQLite，生产迁移至 PostgreSQL）。
  - 大对象/静态资源 → 对象存储。
- **MUST** 生产环境**禁止**用本地文件（`backend/data/*.json`）作为可写业务存储——多实例下不共享、无并发保护。文件仅作只读样本或导出。
- **MUST** 幂等写入：对“写日记”等写操作，用 `request_id`（或 `report_id+date+内容hash`）去重，防止重试/抖动产生重复条目。
- **SHOULD** 明确每类数据的生命周期与 TTL、归档与删除策略（尤其日记等隐私数据）。

## 5. API 设计规范

- **MUST** 统一响应结构：成功/失败一致的外层格式（如 `{status, data, error_code, message}`），**错误码枚举化**、集中定义，前后端共用。
- **MUST** 所有入参经 Pydantic 模型校验；**禁止**信任客户端传入的身份/权限字段。
- **MUST** 破坏性变更走版本化（`/api/v2/...`），旧版保留兼容期。遵循 CLAUDE.md：**任何 analysis type / route / API / domain / rule 变更必须同步更新 `PRD.md`。**
- **MUST** 长耗时/流式接口用 SSE（或 WebSocket），**不复用**同步端点的契约，避免破坏现有集成。
- **SHOULD** 写操作支持幂等键；分页统一 `limit/offset` 或游标，**禁止**无上限返回。

## 6. 数据库规范

- **MUST** 走连接池，async 驱动（如 `asyncpg`/SQLAlchemy async）。**禁止**每请求新建连接。
- **MUST** Schema 变更走**迁移脚本**（Alembic），可版本化、可回滚，**禁止**手改生产表结构。
- **MUST** 所有查询参数化（防注入）；高频查询字段（`report_id`、`user_id`、`created_at`、日记关键词）**必须**建索引。
- **MUST** 事务边界清晰；跨表写操作用事务保证一致性。
- **SHOULD** 慢查询日志与 N+1 检查纳入 review。

## 7. 安全与合规（含疗愈产品特有底线）

- **MUST** **危机干预节点（不可协商的伦理底线，偿还 D6）**：对话链路**必须**前置一个纯规则、零延迟的危机检测器（自伤/自杀/极端情绪关键词与模式）。命中时**立即脱离占星话术**，输出预设的真实支持资源（求助热线等），**并记录事件**。此逻辑独立于 LLM，不得因 LLM 降级而失效。
- **MUST** 遵循产品伦理（源自 memory/product-positioning）：**不贩售恐惧、不做宿命断言、不制造依赖、不给具体医疗/投资建议。** 这些约束**必须**同时体现在 System Prompt 与后端护栏两处（不能只靠 prompt）。
- **MUST** 密钥/凭证**只从环境变量/密钥管理读取**，**严禁**硬编码或提交进 git。`.env*` 必须在 `.gitignore`。
- **MUST** 鉴权集中处理；用户只能访问自己的报告/日记/会话（当前部分端点已按 `user_id` 过滤，**须全量核查**无遗漏）。
- **MUST** 隐私数据（生日、地点、对话、日记）遵循最小化收集、加密传输；提供删除路径。
- **SHOULD** 对外接口限流 + 输入长度上限，防滥用与 prompt 注入。
- **SHOULD** 内容安全：对 LLM 输出做敏感内容兜底过滤。

## 8. 可观测性（线上没有它就是盲飞）

- **MUST** 结构化日志（JSON），每条带 `request_id`/`trace_id`、`user_id`（脱敏）、路径、耗时、状态。**禁止**用 `print` 作为生产日志。（偿还 D9）
- **MUST** 关键指标可采集：QPS、P50/P95/P99 延迟、错误率、LLM 调用耗时与失败率、降级次数、危机命中次数。
- **MUST** 每个请求有贯穿全链路的 `request_id`，日志/错误/追踪可关联。
- **SHOULD** 接入 APM/分布式追踪（OpenTelemetry）；关键异常触发告警。
- **SHOULD** 健康检查端点（`/health` 存活、`/ready` 依赖就绪）供负载均衡与编排使用。

## 9. 性能与容量

- **MUST** 上线前对核心链路（对话、报告生成）做**负载测试**，得出单实例容量与扩容曲线，容量规划有数据支撑而非拍脑袋。
- **MUST** 限流与背压：入口限流（按用户/IP），下游（LLM/DB）过载时快速失败或排队，**禁止**无限堆积拖垮进程。
- **SHOULD** 多级缓存：星盘计算结果、今日星灵、静态目录等按合理 TTL 缓存。
- **SHOULD** 参数（缓冲刷新间隔、超时、连接池大小、限流阈值）**按压测结果调优**，全部可配置，不硬编码。

## 10. 可靠性与韧性

- **MUST** 优雅关闭：`lifespan` 中关闭连接池、flush 日志、取消在途任务。
- **MUST** 关键外部依赖（LLM、DB、Redis）**每一个**都有明确的失败降级路径，任一依赖抖动不导致全站不可用。
- **MUST** 幂等（见 §4）+ 有限重试（见 §3），避免重复副作用与雪崩。
- **SHOULD** 依赖调用加超时是默认动作——**任何没有超时的外部/跨线程调用都视为 bug。**

## 11. 配置与依赖管理

- **MUST** 遵循 12-Factor：配置全部来自环境变量，代码与配置分离，一份镜像跑所有环境。
- **MUST** 依赖**完整且锁定版本**：`requirements.txt`/`pyproject.toml` 列全（含 `pyswisseph`、`httpx` 等），生产用锁文件保证构建可复现。（偿还 D7）
- **MUST** 新增依赖需评估必要性、维护活跃度、许可证，并在 PR 说明。
- **SHOULD** 前后端环境变量在 `.env.example` 中登记，含说明与默认值。

## 12. 测试规范

- **MUST** 新增业务逻辑**必须**带测试；修 bug **必须**先加复现测试。
- **MUST** 分层测试：单元（引擎纯逻辑）／集成（DB、端点）／契约（API 响应结构）。
- **MUST** 引擎核心（dignities、aspects、scoring、firdaria 等）保持可离线单测，不依赖网络/LLM。
- **MUST** 外部依赖（LLM、地理编码）在测试中 mock，**禁止**测试真实打网络。
- **SHOULD** 关键路径覆盖率门槛（建议 ≥ 70%），CI 中强制。
- **SHOULD** 上线前跑负载测试脚本（见 §9）。

## 13. CI/CD 与分支策略

- **MUST** 建立 CI（GitHub Actions），PR **必须**通过：lint + 类型检查 + 测试，否则阻断合并。（偿还 D8）
- **MUST** 主分支（`main`）保护，**禁止**直接推；开发在 feature 分支，经 PR + review 合入 `dev`，再发布。（当前在 `dev`，符合。）
- **MUST** 提交/PR 说明清楚“做了什么、为什么、影响面、是否需同步 PRD”。
- **SHOULD** 部署可回滚；数据库迁移与代码部署解耦、可独立回退。

## 14. 代码质量

- **MUST** 统一格式化与静态检查（`ruff`/`black` + `mypy`），配置入库，CI 强制。
- **MUST** 类型注解齐全（公共函数签名必须有类型）；`from __future__ import annotations` 已是项目惯例，延续。
- **MUST** 遵循 CLAUDE.md：**注释从简，只注释复杂逻辑**；中文占星术语保持一致（庙旺/失势/落陷/法达/飞星…）。
- **SHOULD** 消除重复：两套对话系统的公共能力抽公共模块。
- **SHOULD** 命名清晰、避免魔法数字（阈值、权重集中为常量并注明依据）。

## 15. 领域特有规范（占星引擎）

- **MUST** 五层解读链（行星→星座→宫位→飞星→相位→组合引擎）与古/现双轨的既有架构不得被绕过；新解读逻辑挂进规则层，**不写死个案**。
- **MUST** 领域权重（古占/现占比例，见 CLAUDE.md 表）变更须同步 PRD 并说明依据。
- **MUST** “AI 是糖衣、引擎是炮弹”：LLM 只做表达润色与对话编排，**事实与判断来自引擎**。禁止让 LLM 成为事实来源。
- **SHOULD** 引擎输出与 AI 输出在数据结构上可分离、可分别测试、可分别降级。

## 16. 变更管理与文档

- **MUST** 重大架构/产品决策写入 `memory/`，并更新 `memory/MEMORY.md` 索引（CLAUDE.md Agent Memory 规范）。

## 17. AI-Native 架构规范（Tool Calling / 流式 / LLM 成本与安全）

> 本项目正从"LLM 翻译机"演进为"Agent + Tool Calling"。传统后端规范（§1–§16）不足以约束 LLM 原生链路，本节专治之。条款分三档：**立刻铁律**（现在就生效）／**触发式**（做到某功能才生效）／**预留**（待规模或基建到位，SHOULD）。
>
> **默认工具调用机制：原生 function calling**（DeepSeek/OpenAI 兼容）。故"悬空调用修复"为立刻铁律。流式 XML 为未来可选，不作默认。

### 17.1 Tool Calling（立刻铁律）

- **MUST** **工具超时独立于 LLM 超时**：LLM 生成超时（§3）与工具执行超时是两回事。每个工具（查星盘、查库、检索日记）必须有自己的超时与失败处理；工具内的 DB/星历调用遵循 §2（阻塞调用丢线程池 + 超时）。
- **MUST** **工具输出必须经裁剪/摘要层**：禁止把引擎原始大对象（`service.py` 产出、完整星盘 JSON）直灌 LLM 上下文。工具返回前必须裁剪为结构化结论（参照现有 `evidence[:N]`、`full_text[:N]` 做法），防止耗尽上下文窗口导致模型遗忘核心指令。
- **MUST** **防幻觉红线（延续 §3）**：星盘/事实只能来自工具返回。工具失败时，异常须捕获并**转成自然语言**回灌模型（"我暂时看不清这部分"），System Prompt 必须禁止模型在工具未成功返回时编造数据。
- **MUST** **悬空工具调用修复**（因默认 FC 而生效）：多轮对话中若用户在工具调用未完成时打断，历史消息会缺 `tool_result`。每次请求 LLM 前，**必须**有中间件扫描消息历史，为缺失结果的 tool_call 补占位（如 `[系统提示] 上一次查询被新消息打断`），防止 API 报错或模型逻辑错乱。

### 17.2 流式输出 SSE（立刻铁律 + 预留）

- **MUST** **应用层心跳保活**：SSE/WebSocket 必须实现应用层心跳（如定期 comment 帧），防止 Nginx/网关/代理默认 60s 超时切断长连接。这是"能不能用"，非优化。
- **MUST** **垫场句**（延续 §3）：触发工具/长计算时立即先推共情文本，后台执行，返回后续流，用户永不干等。
- **MUST** **流式中途失败的降级话术**：当部分输出已发送、LLM 中途失败或超时，**必须**有预设的收尾/兜底文案（如"……我这边星图的信号弱了一下，我们缓一缓"），禁止让用户停在半句话或看到报错。
- **MUST** **断连取消**（延续 §3）：客户端断连必须取消后台 LLM 请求，释放连接与配额。
- **SHOULD**（预留）**背压与有界缓冲**：高并发下流式输出必须有有界缓冲；消费过慢时的策略（丢帧/断连/排队）与缓冲大小、刷新间隔**由压测数据确定**，不预先硬编码数值。

### 17.3 LLM 成本与滥用防护（立刻铁律）

- **MUST** **单 user/session Token 硬上限**：必须按 `user_id` 和 `session_id` 统计 token 消耗并设硬上限，超限强制终止本轮。防止 prompt 注入或恶意刷接口造成的"Token 炸弹"与账单失控。
- **MUST** 入口限流（延续 §7 §9）叠加 token 预算，双重防滥用。
- **SHOULD** 记录每次调用的 token、耗时、provider、是否降级（延续 §3），用于成本归因。

### 17.4 数据隐私与合规（立刻铁律）

- **MUST** **日志层 PII 脱敏**：日志中**禁止**出现明文手机号、验证码、完整生日、对话/日记原文。（现存违规：`main.py:2704` 明文 print 验证码——须修。）
- **MUST** **LLM 上下文最小化**：只向 LLM 传引擎所需的结构化结论（落某宫/某星座/某相位），**禁止**传入用户姓名、原始生日字符串等无关 PII。
  - ⚠️ **注意区分**：出生时间、经纬度是**引擎的必需输入**，不是要脱敏的对象；脱敏发生在"进入 LLM 文本上下文"这一层，而非引擎计算层。禁止无脑对全部字段掩码，否则引擎无法工作。
- **MUST** **数据生命周期与被遗忘权**：疗愈数据（对话、日记）必须有明确 TTL / 归档策略，并提供**一键彻底清除**某用户全部对话与日记的接口。这是合规硬要求。
- **MUST** **写操作幂等**（延续 §4）：写日记等用 `request_id` 去重，防重试/抖动产生重复。

### 17.5 AI 质量可观测与评估

- **SHOULD**（待 §8 基建）**AI 专属指标**：将"工具调用失败率""模型触发安全护栏拒答率""回复过短/空回复率""降级次数"作为独立指标，评估 Agent 质量。
- **MUST**（触发式）**Prompt/编排变更必跑黄金集**：当修改星灵/星语者的 System Prompt 或对话编排逻辑时，必须跑一遍对话"黄金测试集"（先小而精，10–20 个代表性场景），防止"对齐税"（修一处、坏多处）。
  - **黄金集必须覆盖安全场景**：危机关键词必须触发干预节点（§7）、工具失败必须不编造数据、越权 report_id 必须被拒。
- **MUST** 引擎层（规则驱动、确定性）保持独立回归测试（延续 §12），与 LLM 输出的 Eval 分离。
