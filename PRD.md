# 占星人生模型 PRD

版本：v0.3  
更新日期：2026-06-12

## 1. 产品定位

### 1.1 产品名称
占星人生模型（Astrology Product System）

### 1.2 产品目标
把原来单点的“法达 K 线工具”升级为可扩展的占星分析产品体系。

产品不再按单一术法组织，而是按用户问题组织：
- 认识自己：本命蓝图
- 看当前阶段：阶段导航
- 看年度重点：年度节奏
- 看内在演化：次限演化
- 看关系互动：关系合盘

### 1.3 核心原则
- 本命定义底盘，不与时间法平级竞争。
- 时间法负责阶段，不直接改写本命结构。
- 触发法负责应期，不负责重建命题。
- 首页展示分析目录，不绑定某一种方法。
- 报告页采用通用壳子，新模型通过注册表和模块扩展。

## 4. 信息架构

### 4.1 前端路由
- `/`
  - 分析目录首页
- `/analysis/:type`
  - 通用分析输入页
- `/reports/:id`
  - 通用报告页
- `/kline`
  - 旧路由兼容别名

### 4.2 页面职责
- 首页：展示分析目录、当前开放能力与规划能力，并可挂载可隐藏的示例命盘入口
- 输入页：承载统一资料输入骨架
- 报告页：按分析类型挂载通用模块和专用模块

### 4.3 示例入口策略
- 首页允许配置一个“小入口”展示标准案例，不打断主 CTA。
- 当前标准案例使用黄金荣，用于展示极端人生样本、阶段法校验与内容深度。
- 黄金荣标准案例数据固定为：`1868-12-14 00:01 / 浙江余姚 / 121°10′E / 30°03′N`。
- 示例入口需支持随时隐藏，避免首页长期被单一案例绑架。
- 点击示例应直接进入报告页，优先让用户先看内容，再决定是否录入自己的资料。

## 5. 技术架构

### 5.1 后端架构
后端采用“分析目录 + 分析编排 + 报告存储”的结构。

#### 已实现
- `GET /api/analysis-types`
  - 返回分析目录
- `POST /api/analyses`
  - 通用分析创建接口
- `GET /api/analyses/{report_id}`
  - 通用报告读取接口
- `POST /api/analyze`
  - 旧接口兼容层，内部转发到 `phase_navigation`
- `GET /api/report/{report_id}`
  - 旧报告接口兼容层

#### 数据存储
报告文件当前保存为统一记录：
- `analysis`
- `request`
- `report`
- `stored_at`

旧版本纯报告 JSON 也兼容读取。

### 5.2 前端架构
前端采用“目录页 + 通用输入页 + 通用报告页”。

#### 已实现
- 首页改为分析目录，不再写死法达入口
- 首页支持可配置示例命盘入口
- 新增通用输入页 `/analysis/:type`
- 结果页切到通用报告路由 `/reports/:id`
- 保留 `/kline` 兼容旧跳转
- 报告页新增本命星盘模块

### 5.3 分析注册表
当前分析类型通过注册表维护，单个分析定义至少包含：
- `key`
- `title`
- `tagline`
- `description`
- `category`
- `status`
- `subjects_count`
- `required_inputs`
- `engines`
- `modules`
- `primary_cta`
- `output_route`

## 6. 输入模型设计

### 6.1 统一输入目标


## 7. 占星解释原则

### 7.1 解释优先级
1. 本命结构
2. 大阶段法
3. 次级时间法
4. 触发与短期波动

### 7.2 当前阶段导航的解释逻辑
- 用本命结构定义长期人格与命题
- 用法达主运 / 子运划分人生阶段
- 用阶段分数和主题模块输出机会、风险和行动建议

### 7.3 后续综合报告原则
- 不把多方法结果简单拼接
- 必须有统一的综合解释层
- 当不同方法冲突时，以本命结构为底层约束

### 7.4 翻译层原则
- 解释不能只停留在“宫位 / 行星基础义”。
- 必须同时提供“成人社会义”和“社会角色义”。
- 同一结构在不同人物上，应能翻译成不同的现实角色，而不是只给抽象关键词。
- 输出语言尽量从术语走向现实语境，减少泛化空话。

### 7.7 高级规则层
- 报告应逐步支持“宫主星飞宫”“同一行星统领多个宫位”“样例级规则链”这类结构化输出。
- 第二层规则优先支持“接纳”“互溶”“转宫”三类关系，用来承接更复杂的现实叙事。
- 高级规则层先服务解释，不先追求术语堆叠。
- 高级规则层应先产出通用规则卡，用统一结构解释命主线、7宫助力、财富结构、幕后代价等主题；历史样例深拆属于其上的专题叙事层。
- 历史样例允许在通用引擎之上增加样例深拆层，用于验证复杂链路是否能落成现实叙事。

### 7.5 本命蓝图内容框架
- 结构层：上升、命主星、主导星、压力点。
- 权力层：资源怎么来、靠谁扩张、冲突怎么发生、隐性控制在哪里。
- 角色层：这张盘更像哪种社会角色，例如组织者、操盘者、经纪人、执行者、统筹者。
- 代价层：这套能力的代价、反噬点，以及晚年更容易如何收束。

### 7.6 历史案例校验原则
- 对历史人物或高说服力案例，报告应允许加入“人生节点校验”模块。
- 节点校验要把重大事件映射到主运 / 子运与本命结构，而不是只列年表。
- 已故人物不应强依赖“当前阶段”叙事，应支持历史时间线模式。

## 8. 视觉与交互方向

### 8.1 当前方向
- 深色天象背景
- 金色作为结构与高价值信息强调色
- 玻璃卡片承载报告模块
- 首页突出“目录”而不是“单模型营销页”

### 8.2 设计要求
- 首页要有产品感，而不是工具感
- 首页示例入口要小而清晰，不打断主路径，并且可以随时隐藏
- 报告页要有阅读秩序，而不是堆字段
- 可视化图表要服务解释，不是装饰
- 本命星盘优先服务阅读和对照，不做脱离解释的炫技展示
